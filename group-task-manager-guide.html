Group Task Manager: How Everything Works

This document explains the current implementation of the Group Task Manager app in this project, including the page structure, the styling system, the JavaScript behavior, and the main limitations.

The app is a lightweight client-side task board. All task data is stored in memory inside the browser while the page is open. There is no database, backend, or local persistence layer.

1. Project Files

index.html
Defines the visible structure of the page, including the hero area, task form, summary metric cards, filter controls, task list, and empty state.
style.css
Contains the full visual design system: layout, colors, typography, spacing, card styling, responsive rules, and interaction states.
app.js
Controls the interactive behavior: adding tasks, deleting tasks, searching, filtering, formatting dates, showing empty states, and updating summary metrics.

2. Page Structure

The HTML is divided into four main UI areas:

  • Hero section: introduces the board and shows the live task count in the summary panel.
  • Composer card: contains the task name input, due date input, and add-task button.
  • Metrics grid: shows counts for total tasks, overdue tasks, upcoming tasks, and tasks without dates.
  • Board card: contains search, filter buttons, the task list, and the empty-state message.

The task list itself is an empty ul in the source HTML. JavaScript generates the list items dynamically every time the UI is rendered.

3. Visual Design System

The CSS uses a warm, paper-like visual direction instead of a generic dark dashboard. The main ideas are:

  • A soft layered background made from gradients, glow elements, and a subtle grid texture.
  • Large editorial typography for section titles and a clean sans-serif for interface text.
  • Card-based layout for the hero, composer, board, and summary panels.
  • Color-coded task states so overdue and undated tasks are easy to distinguish visually.
  • Responsive breakpoints so the layout collapses cleanly on tablets and phones.

How task states are shown visually

  • Normal scheduled tasks use a teal-toned accent.
  • Overdue tasks switch to a warmer alert treatment.
  • Tasks without due dates use a muted neutral style.
  • Search matches are highlighted with a mark background in the task title.

4. JavaScript Data Model

The app keeps all tasks in a single array named tasks. Each task object has the following shape:

{ text: "Task description", dueDate: "YYYY-MM-DD" }

Two additional variables track the current UI state:

  • activeFilter stores which filter button is selected.
  • searchQuery stores the live value of the search box.

5. Main App Flow

Adding a task

  • The user types a task name and optionally chooses a due date.
  • Submitting the form calls addTask(taskText, dueDate).
  • The new task is pushed into the tasks array.
  • The app calls renderTasks() to rebuild the list and update the UI.
  • The inputs are cleared after a successful add.

Deleting a task

  • Each rendered task gets its own delete button.
  • That button removes the task from the array using its original index.
  • After deletion, renderTasks() runs again to refresh the board.

Searching and filtering

  • The search input updates searchQuery on every keystroke.
  • Each filter button updates activeFilter when clicked.
  • getFilteredTasks() combines both conditions to decide which tasks should be visible.

6. Core Functions

formatDueDate
Converts a stored YYYY-MM-DD value into a friendlier localized date string for display.
isOverdue
Compares a task due date to today and returns true only when the date is earlier than the current day.
highlightMatch
Wraps matching search text in a mark tag so search results are visually highlighted.
getFilteredTasks
Creates the task subset that matches the active search query and selected filter.
updateMetrics
Recalculates the four summary cards: total, overdue, upcoming, and no-date counts.
renderTasks
Clears and rebuilds the task list UI, updates the hero count, toggles the empty state, and refreshes the summary metrics.

7. How Rendering Works

The app does not update single list items in place. Instead, it follows a simple full-render pattern:

  • Clear the current task list with taskList.innerHTML = ''.
  • Compute the filtered set of tasks.
  • Create a new li element for each visible task.
  • Add CSS classes such as overdue or no-date based on the task state.
  • Insert the title, due-date badge, and delete button into the item.
  • Append each item to the list.

This approach is straightforward and easy to understand for a small project, even though it is less optimized than a more advanced incremental rendering strategy.

8. Summary Metrics Logic

The summary cards are recalculated every time the board renders:

  • Total: all tasks currently stored in memory.
  • Overdue: tasks with a date earlier than today.
  • Upcoming: tasks that have a due date and are not overdue.
  • No date: tasks whose dueDate value is empty.

The hero panel count changes depending on whether the user is viewing the full board or a filtered result. For example, it can show a value like 2 of 5 tasks during search or filter mode.

9. Current Limitations

  • Tasks are not saved after a browser refresh because there is no local storage or backend.
  • There is no task editing flow yet. Tasks can only be added and deleted.
  • There are no completion states such as done, in progress, or assigned.
  • The app does not validate more advanced date rules beyond checking whether a task is overdue.
  • The UI logic is entirely client-side, so it is suitable for a small demo but not a multi-user production system.

10. Good Next Steps

  • Add localStorage support so tasks persist between page reloads.
  • Support editing an existing task instead of only deleting it.
  • Add completion toggles and separate active/completed sections.
  • Store more metadata such as owner, priority, or notes.
  • Introduce a backend only if the project needs shared team access across devices.

11. In One Sentence

The app is a small browser-based task board that uses static HTML for structure, CSS for the full visual system, and plain JavaScript to manage task data, rendering, search, filters, and summary metrics.
